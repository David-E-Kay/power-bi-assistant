Layer 11. Quick Start

The canonical source is docs/regression-testing-developer-guide.md; this page is its interactive HTML companion.

You're being asked to verify that a model change didn't silently break existing reports. The regression test answers one question: "Do measures still return the same values they did before the change?"

🚦 Before you run any capture — set up your local environment in this order
  1. Open the model in Power BI Desktop. This starts the local Analysis Services instance (msmdsrv.exe) that hosts the model in memory.
  2. Open Tabular Editor 3 and connect it to that model (File → Open → From DB / Workspace mode, then pick the running workspace). TE3 is the runtime that executes the capture script.
  3. Open Claude Desktop or Claude terminal in the relevant project folder. Claude runs from the synced project; the script auto-creates its output folder on first run, so no manual setup is required unless you want to configure the destination directory. If so, open the setup checklist and complete the Output dir card.

The script discovers the AS port automatically by matching TE3's connected database GUID against Power BI Desktop's running workspaces — you don't configure anything by hand. But it cannot do that match if either Power BI Desktop or TE3's connection is missing.

Layer 22. What Is Regression Testing Here?

Why test the semantic model, not the reports?

Power BI reports are visualizations using measures. Ensuring that all measures work correctly and return the same results is the goal of a regression test, but setting up reports to test all of this would be highly manual and subject to human error. Plus, multiple reports use the same measures, so testing the semantic model layer validates every report at once.

The "before vs after" pattern

You capture two snapshots: baseline (today's results) and refactored (after your change). The comparison is value-by-value — same query, same filter context, same measure, different results means a regression candidate.

Template + Injection pattern

The capture script scripts/capture-snapshot.csx is a 900-line read-only template — never edited directly. Each session, Claude copies it to output/{label}.csx and populates the model-specific configuration block, based on your conversation:

  1. The PURPOSE comment
  2. modelName
  3. globalFilters
  4. maxRowsPerContext
  5. testLines block (the test cases)
  6. groupByColumns dictionary (DAX column references)

You don't open or hand-edit this file unless you want to. Claude reads the populated sections back to you so you can review before running.

Memory & timeout safeguards

The capture script ships with three safeguards that will need per-machine tuning. Don't skip past them — these are usually what trip up first-day runs.

  • Pre-run smoke test — before the main loop starts, the script runs a grand-total EVALUATE ROW("r", [Measure]) for every unique measure. Any measure that fails (syntax error, broken dependency, smoke-test timeout) lands on a skip list and is recorded as skipped in every test case for that measure during the main run. This prevents one broken measure from cascading timeouts across the entire suite. Tune the smoke-test budget via SMOKE_TEST_TIMEOUT_MS (default 10s).
  • Memory watchdog — checks system RAM between tests; aborts the run with status aborted_memory if usage stays above MEMORY_THRESHOLD_PCT (default 80%) for several consecutive checks. Prevents your dev box from locking up when a query explodes.
  • Per-query timeout — QUERY_TIMEOUT_MS (default 60s) cancels any individual query that runs too long; the test is recorded as timeout and the run continues with the next test.

Tune these based on your machine and your concurrent local workload:

  • Less RAM, or Power BI Desktop / Docker / IDEs / browsers running alongside? Drop the threshold to 70% so the watchdog trips earlier and protects you from a frozen box. Bump QUERY_TIMEOUT_MS if individual queries are slow because of contention rather than because they're broken.
  • Headroom to spare on a workstation-class machine? Push the threshold up to 90% to avoid false-positive aborts on heavy models.

If a run aborts on memory, see §8 Triage for the recovery sequence.

Layer 36. How the Capture Script Works

Two halves to the lifecycle: Part 1 covers how Claude turns your description of a model change into the populated output/{label}.csx file you'll run; Part 2 covers what that file does once you press F5 in TE3. Neither half asks you to write DAX or C#.

Part 1 — How Claude generates the script from the template

Claude does the authoring; you confirm in chat. The capture template at scripts/capture-snapshot.csx is read-only — Claude never modifies it, only fills in data.

  1. Reads the model schema. Claude pulls the model's tables, columns, measures, relationships, and calc groups from artifacts/model-schema/{model}.md via the powerbi-context-mode skill — even very large models stay out of the context window.
  2. Classifies measures. From your plain-English change description, Claude tiers measures into Tier 1 / 2 / 3 / Skip and applies any plain-English exclusions you give it ("skip all EOP measures", "drop anything _-prefixed", "calc-group helpers", and so on).
  3. Proposes the test matrix. Dimensions, single-dimension vs cross-product contexts, optional globalFilters, optional TOPN row cap (maxRowsPerContext) — Claude proposes, you confirm.
  4. Copies the read-only template scripts/capture-snapshot.csx to output/{label}.csx. The template's DAX-construction logic is verbatim and tested — Claude never writes DAX or C#, only fills in data.
  5. Populates the 6 editable sections of the copy:
    • PURPOSE comment — what model and what change this run validates
    • modelName — written into the snapshot JSON header
    • globalFilters list — KEEPFILTERS expressions applied to every measure
    • maxRowsPerContext — per-context TOPN cap, or 0 for no cap
    • testLines block — one entry per (measure, context-label) pair
    • groupByColumns dictionary — maps each context-label to its fully-qualified DAX columns
  6. Reads the populated file back to you in chat for confirmation before you run it.

What testLines + groupByColumns look like once Claude is done populating them:

testLines.Add("[Measure A]", "by Month and Status");
groupByColumns["by Month and Status"] = new[] {
    "'Date'[Start of Month]",
    "'Table A'[Column A]"
};

Once you've reviewed the populated file in chat and given the green light, open output/{label}.csx in TE3 (with TE3 already connected to your model — see §1's pre-run callout) and press F5. That's the hand-off into Part 2.

Part 2 — How the generated script runs

What happens once you press F5, in order:

  1. Reads configuration. Model name, output directory, safeguard thresholds (QUERY_TIMEOUT_MS, SMOKE_TEST_TIMEOUT_MS, MEMORY_THRESHOLD_PCT), optional globalFilters, optional maxRowsPerContext.
  2. Connects to the model. Reads TE3's Model.Database.Name (a GUID for local PBIP workspaces), scans Power BI Desktop's AnalysisServicesWorkspaces folder for running msmdsrv instances, matches the GUID to a port, and opens a direct ADOMD connection. Both Power BI Desktop AND TE3 must already be open with the model loaded — no other connection mode is supported.
  3. Pre-flight smoke test. For each unique measure, runs a grand-total EVALUATE ROW("r", [Measure]). Any measure that fails (syntax error, broken dependency, smoke-test timeout) lands on a skip list and is recorded as skipped in the main run for that measure — keeps one broken measure from cascading timeouts across the whole suite.
  4. Main capture loop. For each testLines entry, looks up the dimension columns from groupByColumns, builds a SUMMARIZECOLUMNS query around the measure, wraps it in CALCULATE + KEEPFILTERS for any globalFilters, optionally caps rows via TOPN(maxRowsPerContext, …), executes with the per-query timeout, and streams the result to {label}.json per test (so memory stays flat even for big suites).
  5. Memory watchdog runs between tests; aborts cleanly with status aborted_memory if RAM stays above the threshold for several consecutive checks. (See §8 Triage for the recovery sequence.)
  6. Writes supplementary outputs: {label}-testplan.json, {label}-timing.csv, {label}-errors.log, {label}-timeouts.log.
  7. Optional Teams notification on completion if a webhook URL has been set up.

What a single per-test query looks like by the time the script has built it from one testLines + groupByColumns pair plus any globalFilters:

EVALUATE
TOPN(
    5,
    SUMMARIZECOLUMNS(
        'Date'[Start of Month],
        'Table A'[Column A],
        "Result", CALCULATE([Measure A], KEEPFILTERS('Date'[Year] = 2025))
    )
)

You don't write this query — the script generates it for every test case automatically from the data Claude populated in Part 1.

Recovering from a memory-pressure abort

If the run halts with status aborted_memory — the watchdog tripped because system RAM stayed above MEMORY_THRESHOLD_PCT for too many consecutive checks — recover in this order. Don't just restart blindly; you'll likely hit the same wall.

  1. Free up resources first. Close Power BI Desktop, browsers, IDEs, Docker, Slack/Teams, and anything else competing for RAM. On a typical laptop this alone usually unblocks the run.
  2. Raise the threshold if you have headroom. Bump MEMORY_THRESHOLD_PCT in output/{label}.csx (e.g. 80 → 90) to reduce false-positive aborts on heavy models.
  3. Identify the hung query in the logs. Open {label}-timeouts.log and look for the test running just before the abort line. The measure name and context label tell you which test was the culprit.
  4. Quarantine the culprit and re-run. Remove that measure (or just its specific context) from the testLines block — Claude can do this surgically if you paste the log entry — and re-run. The rest of the suite will complete.
  5. Investigate quarantined measures separately. Each removed measure needs follow-up outside the regression test (DAX Studio trace, query plan analysis, refactor). Don't leave them quarantined indefinitely — they're a known-broken signal that the regression itself can't tell you about.

The comparator scripts/compare-snapshots.py needs Python on your PATH. Pick whichever install path matches how you usually manage tools — any of the three works.

Install the official "Python" extension by Microsoft from the VS Code Marketplace. On first run with a .py file open, it offers to download and configure an interpreter for you. No admin rights needed.

Option B — python.org installer

Download from python.org/downloads. The "Install for all users" option requires admin; the per-user default install does not, and is sufficient for our use. Tick "Add Python to PATH" during install.

Option C — winget (PowerShell)

One-liner; runs as your user, no admin prompt:
