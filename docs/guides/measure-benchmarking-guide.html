Layer 11. Quick Start

The canonical source is docs/measure-benchmarking-developer-guide.md; this page is its interactive HTML companion.

You're being asked to identify the slowest measures in a model for optimization prioritization. The benchmark answers one question: "Which measures should I optimize first, and by how much?" It captures timing only — no value validation, no before/after comparison.

🚦 Before you run any benchmark — set up your local environment in this order
  1. Open the model in Power BI Desktop. This starts the local Analysis Services instance (msmdsrv.exe) that hosts the model in memory.
  2. Open Tabular Editor 3 and connect it to that model (File → Open → From DB / Workspace mode, then pick the running workspace). TE3 is the runtime that executes the benchmark script.
  3. Open Claude Desktop or Claude terminal in the relevant project folder. Claude runs from the synced project; the script auto-creates its output folder on first run, so no manual setup is required unless you want to configure the destination directory. If so, open the setup checklist and complete the Output dir card.

The script discovers the AS port automatically by matching TE3's connected database GUID against Power BI Desktop's running workspaces — you don't configure anything by hand. But it cannot do that match if either Power BI Desktop or TE3's connection is missing.

Layer 22. What Is Measure Benchmarking Here?

Why benchmark the semantic model layer?

Power BI report performance has two halves: visual rendering (browser-side) and query execution (semantic-model-side). The slow half is almost always query execution — a measure that takes 8 seconds to evaluate will make every visual that uses it sluggish. Benchmarking the semantic model layer via DAX queries gives you per-measure timing data that maps directly to optimization work.

Manual DAX Studio profiling tests one measure at a time. Running 25–50 measures across multiple filter contexts manually would take hours and produce inconsistent timing (different cache states, different machine loads). The benchmark script automates this — pre-flight smoke test, controlled timing per query, CSV output you can sort.

In addition to the manual and time consuming aspects outlined above, capturing measure performance at the report layer is another non-starter, and introduces additional noise due to the number of visual containers on the page (regardless if they query data or not, such as a text box), which create additional overhead. Simply looking at how long it takes a report page or visual to load is NOT sufficient for accurate benchmarking and optimization prioritization. In fact, Microsoft recommends intentionally limiting the number of visuals on a page (see Microsoft's Power BI optimization guidance).

The "find the slowest" pattern

You define the measure list (which measures to profile) and the context matrix (which filter contexts to evaluate them in). The script runs every (measure × context) combination, writes timing to CSV, and the slowest entries are your optimization candidates. There's no "before/after" — just a single snapshot of current performance.

Single run per test case — no warm/cold averaging

TE3 scripting cannot clear the Analysis Services engine cache between queries. Multiple runs would measure warm-cache performance after the first query, skewing results. A single pass gives the most honest mixed-cache timing — representative of typical report usage where some measures hit the cache and others don't.

Template + Injection pattern

The benchmark script scripts/benchmark-measures.csx is a tested, read-only template — never edited directly. Each session, Claude copies it to output/{label}.csx and populates the 6 configuration sections, based on your conversation:

  1. measures — the list of measure names to profile
  2. singleSliceDimensions — label → DAX column dictionary for one-dimension-at-a-time tests
  3. crossProductColumns — list of DAX columns for the combined cross-product context
  4. crossProductValueFilters — column → values dictionary for TREATAS slicer simulation
  5. globalFilters — KEEPFILTERS expressions applied to every query
  6. maxRowsPerContext — TOPN cap, or 0 for no cap

Note: the pattern uses TREATAS within the constructed DAX queries to replicate the query generated by Power BI visuals.

The script's helpers (smoke test, memory watchdog, ADOMD execution, TREATAS construction, CSV writer, Teams webhook, summary report) are copied verbatim from the template. Claude never modifies them — only fills in data.

Memory & timeout safeguards

The benchmark script ships with a layered safety stack — each layer catches a different failure mode. Don't disable these on first run; they're what keeps a runaway query from locking up your dev box.

  • Pre-flight smoke test — runs a grand-total EVALUATE ROW("r", [Measure]) for every unique measure with a tight timeout. Failures land on a skip list and are recorded as skipped in every test case for that measure during the main run. Tune via SMOKE_TEST_TIMEOUT_MS (default 10s).
  • Per-query timeout — QUERY_TIMEOUT_MS (default 60s) cancels any individual query via cmd.Cancel() on a thread-pool task. The test is recorded as timeout and the run continues.
  • Memory watchdog with debounce — checks RAM every 500 ms during query execution; cancels after 3 consecutive critical readings (≈1.5 s sustained pressure) above MEMORY_THRESHOLD_PCT (default 80%). Between tests, a single critical reading aborts the run with status aborted_memory.
  • Direct ADOMD with cancellation — USE_DIRECT_ADOMD=true (default) uses cancellable threaded execution. false falls back to EvaluateDax() which has no timeout and no cancellability — escape hatch only.
  • Smoke-skip toggle — SKIP_ON_SMOKE_FAILURE=true (default) skips smoke-failed measures in the main run. false attempts every measure regardless and relies on wall-clock + memory watchdogs.

If a run aborts on memory or you see widespread timeouts, see §8 Triage for the recovery sequence.

False Negatives Safeguard

The timing CSV captures both row_count and distinct_values per query. This is not result validation — values aren't checked against a baseline — but it's a quick sanity test on whether the measure actually evaluated under the test's filter context.

If distinct_values = 1 while row_count > 1, the measure returned the same value for every grouping. That usually means the measure is degenerate under that context — missing relationship path, blocked filter propagation, hard-coded constant — and the engine short-circuited to a single value. A fast duration_ms on a degenerate measure is not an optimized query. Open the DAX in DAX Studio before crossing it off the optimization list.

Decision points reference

Phase 1 — Measure selection criteria. Three approaches:

  • Domain description (semantic search): "All measures for [Domain A], [Domain B] costs and counts, and [Domain C] counts, excluding all budget measures and all time intelligence variations"
  • Explicit list: paste measure names directly; Claude validates against the model
  • Hybrid: domain description plus specific additions or removals

Common exclusion categories to clarify: time intelligence variants ("(ly)", "(ytd)", "MoM", "YoY", "vs PY"), budget / planning measures, internal helpers (e.g., _-prefixed).

Phase 2 — Single-slice vs cross-product. Single-slice runs SUMMARIZECOLUMNS('Table'[Column], "Result", [Measure]) per (measure, dimension) — one row per distinct value. Cross-product combines multiple grouping columns with optional TREATAS value filters in one SUMMARIZECOLUMNS — mimics a matrix visual with multiple axes and slicers, the most expensive query shape and where performance issues often hide. Global filters are KEEPFILTERS applied to every query (mimics report-level filters).

If the cross-product row count is very high (>5,000), narrow it with TREATAS value filters or use TOPN — otherwise you'll hit timeouts on every measure regardless of DAX quality.

Layer 36. How the Benchmark Script Works

Two halves to the lifecycle: Part 1 covers how Claude turns your description of what to profile into the populated output/{label}.csx file you'll run; Part 2 covers what that file does once you press F5 in TE3. Neither half asks you to write DAX or C#.

Part 1 — How Claude generates the script from the template

Claude does the authoring; you confirm in chat. The benchmark template at scripts/benchmark-measures.csx is read-only — Claude never modifies it, only fills in data.

  1. Reads the model schema. Claude pulls the model's tables, columns, measures, and relationships from artifacts/model-schema/{model}.md via the powerbi-context-mode skill — even very large models stay out of the context window.
  2. Proposes a measure list grouped by domain. From your plain-English description, Claude semantically interprets which measures fit (e.g., "all project measures" doesn't require the word "Project" in the name — Claude reasons about meaning, not literal substring match) and applies any exclusions you give it.
  3. Proposes the test matrix. Single-slice dimensions, optional cross-product columns and TREATAS value filters, global filters, optional TOPN cap (maxRowsPerContext) — Claude proposes, you confirm.
  4. Copies the read-only template scripts/benchmark-measures.csx to output/{label}.csx. The DAX-construction logic is verbatim and tested — Claude never writes DAX or C#, only fills in data.
  5. Populates the 6 editable sections of the copy:
    • measures — confirmed measure list
    • singleSliceDimensions — label → DAX column dictionary
    • crossProductColumns — DAX columns for the combined context
    • crossProductValueFilters — column → values for TREATAS
    • globalFilters — KEEPFILTERS expressions
    • maxRowsPerContext — TOPN cap (or 0)
  6. Reads the populated file back to you in chat for confirmation before you run it.

What a populated cross-product config looks like once Claude is done:

measures.Add("[Measure A]");
measures.Add("[Measure B]");

singleSliceDimensions["by_col_a"] = "'Table A'[Column A]";
singleSliceDimensions["by_month"] = "'Date'[Month]";

crossProductColumns.Add("'Table A'[Column A]");
crossProductColumns.Add("'Date'[Month]");
crossProductColumns.Add("'Table C'[Column C]");

crossProductValueFilters["'Table C'[Column C]"] =
    new List<string> { "Value 1" };

globalFilters.Add("'Date'[Start of Year] = DATE(2025, 1, 1)");
maxRowsPerContext = 0;

Once you've reviewed the populated file in chat and given the green light, open output/{label}.csx in TE3 (with TE3 already connected to your model — see §1's pre-run callout) and press F5. That's the hand-off into Part 2.

Part 2 — How the generated script runs

What happens once you press F5, in order:

  1. Reads configuration. Output directory, safety thresholds (QUERY_TIMEOUT_MS, SMOKE_TEST_TIMEOUT_MS, MEMORY_THRESHOLD_PCT), and the 6 populated sections from Part 1.
  2. Connects to the model. Reads TE3's Model.Database.Name (a GUID for local PBIP workspaces), scans Power BI Desktop's AnalysisServicesWorkspaces folder for running msmdsrv instances, matches the GUID to a port, and opens a direct ADOMD connection. Both Power BI Desktop AND TE3 must already be open with the model loaded — no other connection mode is supported.
  3. Pre-flight smoke test. For each unique measure, runs a grand-total EVALUATE ROW("r", [Measure]) with a tight per-measure timeout. Failures land on an in-memory skip list and are recorded as skipped in every test case for that measure during the main run.
  4. Main benchmark loop. Builds the test matrix: 1 grand_total per measure + N single-slice contexts + 1 cross-product context (if configured). For each test case, builds a SUMMARIZECOLUMNS query (with TREATAS args for cross-product, KEEPFILTERS for globals, optional TOPN), runs on a thread-pool task with cmd.Cancel()-based wall-clock timeout, and writes {test_id, measure, context, status, row_count, duration_ms, distinct_values} to the timing CSV.
  5. Memory watchdog runs during each query (3-poll debounce) and between tests (no debounce). Mid-query trip cancels just that query (status: timeout); between-test trip aborts the run (status: aborted_memory).
  6. Writes supplementary outputs: {label}-config.csv (filter context audit), {label}-testplan.json (manifest), {label}-timeouts.log (only if any timeouts/smoke failures), {label}-errors.log (only if any errors).
  7. Optional Teams notification on completion if a webhook URL has been set up. Info() popup shows Top 10 Slowest (ok-only), smoke-skipped measures, and Timed Out Queries.

What a single cross-product test query looks like by the time the script has built it:

EVALUATE
SUMMARIZECOLUMNS(
    'Table A'[Column A],
    'Date'[Month],
    'Table C'[Column C],
    TREATAS({"Value 1"}, 'Table C'[Column C]),
    "Result",
        CALCULATE(
            [Measure A],
            KEEPFILTERS('Date'[Start of Year] = DATE(2025, 1, 1))
        )
)

You don't write this query — the script generates it for every test case automatically from the data Claude populated in Part 1.

Layer 38. Triage: Reading the Results

Status values in the timing CSV

StatusMeaning
okQuery completed successfully — duration_ms is real timing
errorQuery errored (DAX reference error, missing relationship, TREATAS value typo) — see {label}-errors.log for full message
timeoutQuery was cancelled — either wall-clock OR sustained memory pressure. Distinguish via Type: tag in {label}-timeouts.log
skippedMeasure failed pre-flight smoke test. One row per dimension permutation with duration_ms = 0
aborted_memoryThe whole run was aborted by the between-test memory check before this test ran. duration_ms = 0

Reading the timeouts log Type tag

TypeWhat happened
query_timeoutWall-clock QUERY_TIMEOUT_MS expired. duration_ms ≈ QUERY_TIMEOUT_MS. Paste DAX into DAX Studio for triage
memory_watchdog3 consecutive critical RAM readings during query. duration_ms ≈ 1.5–3 s but status = timeout
smoketest_timeoutPre-flight EVALUATE ROW("r", [Measure]) exceeded SMOKE_TEST_TIMEOUT_MS. The measure is fundamentally broken or unbounded
smoketest_errorPre-flight smoke test threw a DAX error. Likely a missing dependency or syntax issue
query_errorMain-loop query threw a DAX error

Triage philosophy: runaways are filtered (smoke test), then timeouts are data. A measure that times out under a rich cross-product context but passes single-slice IS the timing data you wanted — it tells you the measure is pathologically slow under that filter combination, exactly what report users would experience.

Optimization prioritization framework

Sort by duration_ms descending. For each candidate, weigh these factors:

FactorWeightAssessment
Query timeHighAbsolute duration. Anything >5s is a candidate; >10s is urgent
User visibilityHighIs this measure on a frequently-used report page?
Context sensitivityMediumSlow only in cross-product? Likely a context transition or cardinality explosion
Measure family sizeMediumIs this a base measure that 10+ derived measures reference? Optimize the base, all dependents improve
Optimization feasibilityMediumKnown anti-patterns visible (nested SUMX, FILTER on table, redundant CALCULATE)?
Row count ratioLowHigh duration ÷ low row count = expensive per-row calculation

Recovering from a memory abort

If the run halts with status aborted_memory — the watchdog tripped because RAM stayed above threshold for too many consecutive checks — recover in this order. Don't restart blindly; you'll likely hit the same wall.

  1. Free up resources first. Close Power BI Desktop, browsers, IDEs, Docker, Slack/Teams. (You'll re-open PBI Desktop and TE3 to re-run.)
  2. Raise the threshold if you have headroom. Bump MEMORY_THRESHOLD_PCT in output/{label}.csx (e.g. 80 → 90).
  3. Identify the culprit. Open {label}-timeouts.log and look for the test running just before the abort line — that's the query that drove RAM up.
  4. Quarantine and re-run. Remove that measure (or that specific context) from output/{label}.csx. Claude can do this surgically if you paste the log entry. Re-run; the rest will complete.
  5. Investigate the quarantined measure separately in DAX Studio with Server Timings. Don't leave it quarantined indefinitely — it's a known-broken signal that the benchmark itself can't tell you about.

Layer 39. Glossary

TermDefinition
smoke testPre-flight EVALUATE ROW("r", [Measure]) per measure with a tight timeout. Catches broken or runaway measures before the main loop
single-sliceA test case where the measure is grouped by exactly one dimension column
cross-productA test case where the measure is grouped by multiple columns at once, optionally constrained by TREATAS — mimics a matrix visual with multiple axes and slicers
global filterA KEEPFILTERS expression applied to every test case — mimics a report-level filter
memory watchdogThe script's RAM monitor that cancels queries (during) or aborts the run (between) when usage exceeds MEMORY_THRESHOLD_PCT
debounceThe 3-consecutive-poll requirement before the mid-query watchdog cancels — prevents false aborts on transient spikes
TREATASDAX function that constrains a column to specific values inside SUMMARIZECOLUMNS — mirrors slicer behavior in visual queries
SE / FEStorage Engine / Formula Engine. SE is fast and parallel; FE is single-threaded and the usual bottleneck
statusThe outcome column in the timing CSV: ok, error, timeout, skipped, aborted_memory
Type tagThe classification in {label}-timeouts.log: query_timeout, memory_watchdog, smoketest_timeout, smoketest_error, query_error
