# Spec: TE-Free Regression Testing & Measure Benchmarking

**Date:** 2026-06-10
**Branch:** `feature/te-free-capture`
**Status:** Approved for planning

## Problem

The two core IP workflows — **regression testing** (`scripts/capture-snapshot.csx`) and
**measure benchmarking** (`scripts/benchmark-measures.csx`) — currently require Tabular
Editor 3 as their execution host. TE contributes nothing essential: with
`useDirectAdomd=true` (the default), all query execution is plain ADOMD.NET against the
local `msmdsrv` instance. TE provides only (a) the C# script host, (b) `Model.Database.Name`
for port-discovery matching, and (c) the `Info()` popup. The TE dependency forces a manual
"open TE3, run script" step and couples the IP to a third-party tool.

## Goal

Port both workflows to Python so Claude Code can run them end-to-end with zero manual
steps and no Tabular Editor dependency. TE remains in use for model-mutation work
(refactor-strategy C# codegen) — that is out of scope here.

## Decisions (confirmed with user 2026-06-10)

1. **Runtime:** Python + pythonnet, loading Microsoft's ADOMD.NET client DLL.
   Verified on this machine: Python 3.14.5; `pythonnet 3.1.0` resolves cleanly;
   ADOMD DLL available at `C:\Program Files\DAX Studio\bin\Microsoft.AnalysisServices.AdomdClient.dll`
   (PBI Desktop is the Store install — its copy is not readable).
2. **Template philosophy preserved:** the stable, tested logic lives in a Python runner
   (never modified per session); the per-session artifact becomes a **JSON config file**
   (pure data) instead of code blocks injected into a `.csx` copy.
3. **.csx templates retired at the end** — removed only after live parity validation
   passes. Git history preserves them.
4. **Live validation** against the user's real model, open in PBI Desktop.

## Architecture

```
scripts/pbi_capture/            # stable engine package (the "template")
  __init__.py
  clr_boot.py                   # pythonnet runtime init + ADOMD DLL discovery/load
  discovery.py                  # msmdsrv workspace scan, port-file parse, catalog enumeration, connection resolution
  watchdog.py                   # real total-RAM (GlobalMemoryStatusEx) + msmdsrv/self working-set check
  executor.py                   # execute_dax(): query thread + 500ms poll + cmd.Cancel() + manual reader
  daxgen.py                     # DAX construction (capture + benchmark forms) — pure, no CLR
  serialization.py              # value sentinels, streaming snapshot writer, timing CSV, testplan, logs — pure
  config.py                     # config dataclasses, JSON load, env overrides, validation — pure
  runner.py                     # run_capture(cfg) / run_benchmark(cfg) orchestration
scripts/capture_snapshot.py     # CLI: python scripts/capture_snapshot.py --config output/<label>.config.json
scripts/benchmark_measures.py   # CLI: python scripts/benchmark_measures.py --config output/<label>.config.json
tests/                          # pytest; live tests gated by PBI_LIVE=1
requirements-dev.txt            # pythonnet, pytest, openpyxl
```

Per-session artifacts (generated by the skills, written to `output/`):
- `output/{label}.config.json` — capture or benchmark config (schema below)

## Compatibility contracts (MUST hold)

1. **Snapshot JSON schema identical to capture-snapshot.csx v9** so
   `scripts/compare-snapshots.py` works unchanged: header keys
   (`snapshot_version`, `model_name`, `captured_at`, `label`, `global_filters`,
   `max_rows_per_context`, `query_timeout_ms`), `results` keyed by test id with
   per-status fields (`ok`: row_count/columns/rows/duration_ms; `timeout`:
   timeout_limit_ms/duration_ms/dax/error; `error`: error/duration_ms; `skipped`:
   skip_reason/duration_ms; `aborted_memory`: duration_ms), `summary` block.
   Value sentinels `__BLANK__` / `__NaN__` / `__INF__`; doubles rounded to 4 decimals;
   booleans serialized as `"True"`/`"False"` strings (C# `ToString()` parity).
2. **DAX construction semantics identical** to the .csx templates: bare table
   expressions (`EVALUATE` added at the call site), `BuildMeasureRef` KEEPFILTERS
   wrapping, single-dimension / cross-product (`|` split) / TOPN-per-group GENERATE,
   benchmark ROW/CALCULATETABLE grand total, TREATAS arg formatting (raw passthrough
   for `Func(`-like and numeric values, quoted otherwise), no IGNORE().
3. **Output file set identical:** `{label}.json`, `{label}-testplan.json`,
   `{label}-timing.csv`, `{label}-errors.log`, `{label}-timeouts.log` (with the same
   `Type:` tags: memory_watchdog | query_timeout | smoketest_timeout | smoketest_error
   | query_error), benchmark `{label}-config.csv`. Same log line formats (triage docs
   depend on them).
4. **Env var overrides preserved:** `SNAPSHOT_LABEL`/`BENCHMARK_LABEL`, `MODEL_NAME`,
   `DIAGNOSTIC_MODE`, `OUTPUT_DIR`, `TEAMS_WEBHOOK_URL`, `QUERY_TIMEOUT_MS`,
   `SMOKE_TEST_TIMEOUT_MS`, `MEMORY_THRESHOLD_PCT`, `SKIP_ON_SMOKE_FAILURE`,
   `CONNECTION_STRING`. Precedence: CLI flag > env var > config file > default.
5. **Behavioral parity:** pre-flight smoke test (`EVALUATE ROW("r", <ref>)`),
   per-query wall-clock + memory watchdog with 3-poll (1.5s) debounce and query
   interrupt (`cmd.Cancel()` then a `conn.Dispose()` backstop — see deviations),
   between-test memory check (capture: 1s settle window; benchmark:
   immediate abort), diagnostic mode capping at 8 tests, streaming flush per test
   case, Teams Adaptive Card webhook, benchmark distinct_values + false-fast
   detection + Top-10 report.

## Deliberate deviations from the .csx templates (document in code comments)

| Deviation | Rationale |
|---|---|
| `USE_DIRECT_ADOMD` removed | Direct ADOMD is the only mode; the `EvaluateDax()` fallback was TE-only and documented as "Avoid". |
| Memory watchdog uses real total RAM via `GlobalMemoryStatusEx` | The hard-coded 16 GB denominator existed only because the TE sandbox blocked P/Invoke. Python has ctypes. `MEMORY_THRESHOLD_PCT` semantics unchanged. |
| Watchdog sums **python** + msmdsrv working sets (was TE3 + msmdsrv) | Same intent: capture host + engine. |
| Catalog match no longer uses TE's `Model.Database.Name` | Auto-discovery: scan workspace port files, enumerate catalogs per instance; exactly one running instance → auto-select its single catalog; multiple → fail with an actionable list (use `--port` or `CONNECTION_STRING`). |
| Default `OUTPUT_DIR` is `output/regression` / `output/benchmark` (repo-local) | Was `Desktop\PBI-Regression`. Project convention: deliverables in `output/`. Env/config override unchanged. |
| Summary report also written to `{label}-summary.txt` | Replaces the `Info()` popup; enables `powerbi-context-mode` analysis. Still printed to stdout. |
| Measure names in config are bare (no brackets); validation rejects `[`/`]` | The .csx comment examples were inconsistent with the code (which adds brackets). Config validation makes this unambiguous. |
| `|`-separated cross-product columns are stripped of surrounding whitespace | Harmless robustness; the C# split kept spaces. |
| `validate_capture` hard-rejects `max_rows_per_context != 0` (the .csx defaulted to 0 but allowed any value, and its comments *recommended* 3–5 for regression) | Regression diffs per-row values: a TOPN cap silently truncates dimension combinations (a delta in a dropped row is an undetectable false-pass) and the generated `TOPN(N, …)` has no `ORDER BY`, so the kept row set is unstable across the baseline vs refactored captures. The `.csx`'s only rationale for a cap was capture speed on high-cardinality dims — that is a *benchmark* concern (timing, not values); `validate_benchmark` still allows the cap. Fast capture smoke is available via `diagnostic_mode` (caps test count, not rows). Decided with the user 2026-06-14. |
| Query interrupt is `cmd.Cancel()` → short grace → `conn.Dispose()` backstop (the .csx relied on `cmd.Cancel()` alone) | Spike finding (2026-06-11): `cmd.Cancel()` interrupts SE-bound queries (~0.04s, the common slow-measure case) but is a no-op for pure formula-engine materializations (90s+, no effect). Dropping the connection interrupts ANY query (~0.03s — kills the socket; the server cancels the orphaned session), giving a clean exit with no leaked threads. Fresh-connection-per-query makes discarding a timed-out connection free, so there is no leak path. Follow-on (2026-06-13 live finding): `cmd.CommandTimeout` is set to `0` (not the per-query timeout) — a server-side CommandTimeout starts a concurrent engine abort that blocks `conn.Dispose()` ~30s on pure-FE queries; the Python watchdog is the sole timeout enforcement. |

## CLI exit codes

- `0` — run completed (individual test errors/timeouts are data, not failures)
- `2` — fatal: invalid config, no msmdsrv instance found, ambiguous instance, CLR/DLL load failure

## Known risks & mitigations

| Risk | Mitigation |
|---|---|
| pythonnet GIL blocks the watchdog thread while the query thread is inside a blocking `ExecuteReader` | **Spike finding (2026-06-11): NOT an issue — the GIL IS released during `ExecuteReader` (the watchdog thread runs throughout).** The real gap was cancellation semantics, not the GIL: `cmd.Cancel()` works for SE-bound queries but not pure-FE ones. Adopted `conn.Dispose()` backstop (see deviations) instead of the worker-subprocess contingency, which is no longer needed. |
| DAX Studio's ADOMD DLL is .NET (Core)-flavored and won't load under the netfx CLR | **Spike finding (2026-06-11): opposite of expectation — coreclr fails at conn.Open() with `FileNotFoundException: System.Configuration.ConfigurationManager`; netfx succeeds.** Default runtime is `netfx`. `ADOMD_DLL_PATH` and `PYTHONNET_RUNTIME` env overrides supported. |
| `System.Decimal` → float conversion precision | compare-snapshots.py tolerance is 1e-4; values rounded to 4 decimals on both sides. Documented. |
| Multiple PBI Desktop instances open during a run | Fail fast with port+catalog list; `--port` / `CONNECTION_STRING` disambiguate. Same fundamental constraint existed in the .csx (matched via TE's DB name). |

## Non-goals

- XMLA/Fabric AAD auth flows (CONNECTION_STRING passthrough only, as today)
- Model-mutation workflows (refactor-strategy C# codegen stays on TE)
- Rewriting `compare-snapshots.py` (works unchanged by contract #1)
- Regenerating the HTML guide bundles in `docs/` (`*-guide.html`, `power-bi-guides-bundle.html`)
  — deferred to a follow-up; the markdown developer guides ARE updated in this plan
- Multi-run cache-warm benchmarking (single-pass semantics preserved)

## Acceptance criteria

1. With the user's model open in PBI Desktop, `python scripts/capture_snapshot.py
   --config <cfg>` produces a snapshot JSON that `compare-snapshots.py` accepts;
   capturing the same model twice and comparing yields **zero value deltas, exit 0**.
2. A deliberately slow query is cancelled by the wall-clock watchdog within
   `timeout + 15s`, recorded as `status:"timeout"` with a `Type: query_timeout` log entry.
3. `python scripts/benchmark_measures.py --config <cfg>` produces `{label}-timing.csv`
   with `distinct_values`, a config CSV, and a Top-10 report.
4. Unit suite (`pytest -m "not live"`) passes with no PBI Desktop running.
5. Both SKILL.md files, CLAUDE.md, knowledge-index.md, powerbi-context-mode SKILL.md,
   and both markdown developer guides reference only the Python workflow.
6. `scripts/capture-snapshot.csx` and `scripts/benchmark-measures.csx` are removed.
